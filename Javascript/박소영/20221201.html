  • 호이스팅에 대해 설명해주세요. (변수/함수 호이스팅, var/let/const)
    • 호이스팅이란
      • 코드 실행 전 선언문이 해당 스코프의 최상단으로 끌어 올려진 것처럼 동작하는 현상
      • JS 엔진은 코드 실행 전 평가 과정을 거치면서 모든 선언문을 스코프에 등록함 (실행 컨텍스트) 즉, 코드 실행 전 선언이 먼저 메모리에 저장됨 ⇒ 이미 변수/함수 선언이 저장되어있기 떄문에 참조/호출이 먼저 나와도 오류 없이 동작
      • JS의 모든 선언문에는 호이스팅이 일어남
      • 호이스팅은 스코프 단위로 동작함

    • 변수 호이스팅
      • 변수는 3단계에 걸쳐 생성됨 (선언 → 초기화 → 할당)
      • var
        • 선언 단계와 초기화 단계가 한번에 진행되어 undefined로 값이 초기화됨
        • 변수 선언문 이전에 참조 가능함 (undefined 반환)
      • let / const
        • let, const, class를 이용한 선언문은 호이스팅이 발생하지 않는 것처럼 동작함
        • 선언 단계와 초기화 단계가 분리되어 진행됨 런타임 이전에 선언 단계가 실행되지만 초기화 단계는 선언문에 도달했을 때 실행됨
        • 즉, 호이스팅은 되지만 초기화되지 않은 상태로 선언만 메모리에 저장됨
        • 스코프의 시작에서 변수의 선언까지 일시적 사각지대(TDZ)에 빠지기 때문
        • 초기화되지 않으면 변수 참조가 불가능하기 때문에 참조 에러가 발생함

    • 함수 호이스팅
      • 함수 호이스팅 : 함수 선언문이 코드의 선두로 끌어 올려진 것처럼 동작하는 현상
      • 함수 선언문으로 정의한 함수와 함수 표현식으로 정의한 함수는 생성 시점이 다름

      • 함수선언문
        • 런타임 이전에 JS 엔진에 의해 선언이 먼저 실행됨 (함수 객체 먼저 생성됨)
        • 런타임에는 이미 함수 객체가 생성되어 있고 함수명과 동일한 식별자에 할당까지 완료한 상태
        • 때문에 함수 선언문 이전에 함수 참조 가능하며 호출도 가능함
        • var 변수는 undefined로 초기화되지만, 함수 선언문은 함수 객체로 초기화됨
      • 함수표현식
        • 변수에 할당되는 값이 함수 리터럴인 문
        • 변수 선언은 런타임 이전에 실행되어 undefined로 초기화되지만 변수 할당문의 값은 런타임에 평가됨
        • 따라 함수 표현식의 함수 리터럴도 런타임에 실행되고 평가되어 함수 객체 생성됨
        • 즉, 함수 호이스팅이 아닌 변수 호이스팅 발생
      • class 선언식
        • let, const 처럼 호이스팅은 일어나지만 초기화되지 않은 TDZ 상태 유지됨
      • class 표현식
        • 함수표현식처럼 변수 호이스팅이 일어나며 class 객체는 런타임에 생성됨

  • 클로저란 어떤 것이고 클로저의 원리는 무엇인가요? 고차함수란 무엇인가요?
    • 클로저란
      • 함수형 프로그래밍 언어에서 사용되는 중요한 특성
      • JS 고유 개념은 아니기 떄문에 클로저의 정의가 ECMAScript 사양에 등장하진 않음
      • 함수와 그 함수가 선언된 렉시컬 환경과의 조합 -MDN
      • 상태를 안전하게 은닉하고 특정 함수에게만 상태 변경을 허용하기 위해 사용 (캡슐화, 정보은닉)
      • 외부 함수보다 중첩 함수가 더 오래 유지되고, 이미 생명 주기가 종료한 외부 함수의 변수를 참조하는 중첩 함수를 클로저라고 부름

    • 클로저의 원리
      1. 중첩 함수가 평가되어 함수 객체가 생성될 때 내부 슬롯 [[Environment]]에는 현재 실행 중인 외부 함수 실행 컨텍스트의 렉시컬 환경의 참조가 저장됨 (= 상위 스코프) (함수는 렉시컬 스코프를 따르는 언어이기 때문에 정의된 위치에 따라 상위 스코프 정적으로 결정됨)
      1. 외부 함수의 실행이 종료되면 중첩 함수를 반환하면서 외부 함수의 생명 주기가 종료됨 즉, 외부 함수의 실행 컨텍스트가 실행 컨텍스트 스택에서 제거됨
      1. 하지만 외부 함수의 렉시컬 환경까지 소멸되진 않음 가비지 컬렉터는 어떤 값을 참조하는 변수가 하나라도 있다면 수집 대상에 포함하지 않기 떄문 중첩 함수의 내부 슬롯 [[Environment]]이 외부 함수의 렉시컬 환경을 참조하고 있음
      1. 외부 함수가 반환한 중첩 함수를 호출하면 중첩 함수의 실행 컨텍스트가 생성되고 실행 컨텍스트 스택에 푸시됨 그리고 렉시컬 환경의 외부 렉시컬 환경에 대한 참조에는 중첩 함수 객체의 [[Environment]] 내부 슬롯에 저장되어있는 참조값이 할당됨
      1. 중첩 함수는 외부 함수보다 더 오래 생존함. 이때 외부 함수의 생존여부와 상관없이 상위 스코프 식별자를 참조할 수 있고 식별자의 값을 변경할 수도 있음

    • 고차 함수
      • 함수형 프로그래밍에서 자주 사용되는 개념
      • 함수를 인자로 받거나 함수를 출력으로 반환하는 함수
      • Array.prototype.map, filter, reduce가 빌트인 고차함수
      • 어떤 고차 함수에 함수를 인자로 전달하고, 고차 함수는 함수 자체를 리턴하는 경우, 함수의 인자로 전달되는 함수를 콜백 함수(callback function)이라고 함

  • 프로토타입은 무엇이고 프로토타입체인은 무엇인가?

    프로토타입

    • 다른 객체의 원형이 되는 객체
    • JS는 프로토타입을 기반으로 상속을 구현함 ⇒ 기존의 코드를 재사용함으로 불필요한 중복 코드를 제거함
    • 생성자 함수가 생성할 모든 인스턴스가 공통적으로 사용할 프로퍼티나 메서드를 프로토타입에 미리 구현해 놓음으로써 상위 객체인 프로토타입의 자산을 공유해 사용
    • 모든 객체는 __proto__ 접근자 프로퍼티를 통해 자신의 프로토타입에 간접적으로 접근 가능함
    • prototype 프로퍼티는 생성자 함수가 생성할 인스턴스의 프로토타입을 가리킴 화살표 함수와 ES6 메서드 축약 표현으로 정의한 메서드는 prototype 프로퍼티를 소유하지 않음
    • 프로토타입 객체는 생성자 함수가 생성되는 시점에 더불어 생성됨
    • 프로토타입 상속은 고전적인 상속을 흉내내지만 실제로 상속보단 위임에 가까움

    프로토타입 체인

    • 프로토타입 체인은 상속과 프로퍼티 검색을 위한 메커니즘
    • JS는 객체의 프로퍼티에 접근하려고 할 대 해당 객체에 접근하려는 프로퍼티가 없으면 __proto__ 접근자 프로퍼티가 가리키는 링크를 따라 자신의 부모 역할을 하는 프로토타입의 프로퍼티를 순차적으로 검색함
    • 이를 프로토타입 체인이라고 함
    • 프로토타입 체인은 JS가 객체지향 프로그래밍의 상속을 구현하는 메커니즘임
    • Object.prototype을 프로토타입 체인의 종점이라고 함
    • 스코프 체인 - 식별자 검색 프로토타입 체인 - 프로퍼티 검색
    • 스코프체인과 프로토타입 체인은 별도로 동작하는게 아니라 서로 협력해 식별자와 프로퍼티를 검색

  • 실행 컨텍스트와 스코프 체인은 무엇인가?
    • 실행 컨텍스트
      • JS의 동작원리를 담고 있는 핵심 개념
      • 코드가 실행되려면 스코프, 식별자, 코드 실행 순서등의 관리가 필요한데 이 모든 것을 관리하는 것이 실행 컨텍스트.
      • 식별자를 등록하고 관리하는 스코프와 코드 실행 순서 관리를 구현한 내부 메커니즘으로, 모든 코드는 실행 컨텍스트를 통해 실행되고 관리됨
      • 식별자와 스코프는 실행 컨텍스트의 렉시컬 환경으로 관리하고, 코드 실행 순선느 실행 컨텍스트 스택으로 관리함

    • 스코프 체인
      • 스코프는 함수의 중첩에 의해 계층적인 구조를 갖음 스코프 체인은 렉시컬 환경이 계층적 구조로 연결된 것 (렉시컬 환경 = 스코프를 구분해 식별자를 등록하고 관리하는 저장소 역할)
      • 모든 스코프는 하나의 계층적 구조로 연결되며, 모든 지역 스코프의 최상위 스코프는 전역 스코프
      • 이렇게 스코프가 계층적으로 연결된 것을 스코프 체인이라고 함
      • 변수 참조 시 JS 엔진은 스코프 체인을 통해 변수를 참조한느 코드의 스코프에서 시작해 상위 스코프 방향으로 이동하며 선언된 변수를 검색함 이를 통해 상위 스코프에서 선언한 변수를 하위 스코프에서도 참조 가능
      • 상위 스코프에서 유효한 변수는 하위 스코프에서 자유롭게 참조 가능하지만, 하위 스코프에서 유효한 변수를 상위 스코프에서는 참조 불가능함

  • 이벤트 관리 방법에 대해 설명하세요. 보통 어떤 방식으로 이벤트를 설계하나요? (캡처링,버블링,등록,해제,위임 등)
    • 이벤트 핸들러 : 이벤트가 발생했을 때 호출될 함수
    • 이벤트 핸들러 등록 : 이벤트가 발생했을 떄 브라우저에게 이벤트 핸들러의 호출을 위임하는 것
    • 이벤트 리스너
      • 특정 이벤트가 발생했을 때 그것을 감지하는 역할
      • 더 상위 개념으로는 이벤트 핸들러 (이벤트를 감지하고 다루는 역할) 가 있는데 최근에는 이벤트 리스너 방식으로 주로 사용
      • $increaseButton.addEventListener('click', onIncreaseCountHandler);
    • 이벤트 흐름 : 이벤트가 발생하는 순서
      • 이벤트가 흐르는 방식은 2가지가 있음
      • 이벤트 버블링 : 자식의 이벤트가 부모에게도 전파됨
      • 이벤트 캡처링 : 부모의 이벤트가 자식에게 전파됨
    • 이벤트 등록
      • addEventListener
    • 이벤트 해제
      • removeEventListener
    • 이벤트 위임
      • 에빈트 리스너가 실행할 작업을 부모 요소에게 위임하는 것
      • 이벤트 리스너 지정하는 요소가 많을 수록 페이지 실행속도 느려짐 따라 이벤트 흐름을 사용하면 효과적으로 이벤트 관리 가능 이벤트는 target 엘리먼트를 포함하고 있는 부모 요소에도 영향을 미치기 떄문에 자식을 포함하는 부모 요소에 이벤트 리스너를 지정해 관리함

  • 동기와 비동기에 관해 설명해주세요. JS에서 비동기는 어떻게 처리하나요? (callback, promise, async/await, 비동기처리의 특성과 에러처리 등)
    • 동기와 비동기
      • 동기
        • 순차적/직렬적으로 테스크 수행
        • 요청을 보낸 후 응답을 받아야지만 다음 동작 이루어짐
        • 순차적으로 실행되므로 어떤 작업이 수행중이라면 뒤의 작업은 대기함
        • 블로킹(작업 중단) 발생
      • 비동기
        • 병렬적으로 테스크 수행
        • 현재 작업의 종료여부와 무관하게 다음 작업 실행함
        • 동기방식과 달리 완료 순서 보장되지 않음
        • 블로킹 발생하지 않음

    • JS에서의 비동기 처리

      JS에서의 비동기 처리는 Promise, callback, async/await, 제너레이터를 사용

      • callback
        • 콜백 함수 사용 시 특정 로직 끝났을 때 원하는 동작 실행 가능함
        • 콜백 안에 콜백을 계속 무는 형식으로 작성해야 함
        • 비동기 처리를 위해 콜백 함수를 연속해서 사용 ⇒ 콜백 지옥 일어남
        $.get('url', function(response) {
	parseValue(response, function(id) {
		auth(id, function(result) {
			display(result, function(text) {
				console.log(text);
			});
		});
	});
});

      • Promise
        • JS 비동기 처리에 사용되는 객체
        • 주로 서버에서 받아온 데이터를 화면에 표시할 때 사용함
        • 세 가지 상태 가짐
          • Pending (대기) : 비동기 처리 로직이 아직 완료되지 않은 상태
          • Fulfilled (이행) : 비동기 처리가 완료되어 프로미스가 결과 값을 반환한 상태
          • Rejected (실패) : 비동기 처리가 실패하거나 오류가 발생한 상태
        • new Promise() 메서드 호출 시 콜백 함수를 선언 가능하고, 콜백 함수의 인자는 resolve, reject
        • 이행 상태가 되면 then()을 이용해 처리 결과값 받을 수 있음. 실패 상태가 되면 catch()로 실패 처리의 결과값을 받을 수 있음
        • then() 메서드 호출 시 새로운 프로미스 객체가 반환되어 여러 개의 프로미스를 연결 가능

          • catch()를 이용 (권장)

      • typescript, react의 JSX도 브라우저에서 동작하도록 만듬 (트랜스파일)
      • babel은 문법을 변환해 JS로 변환하는 트랜스파일러 역할만함 (ES6 → ES5 이하의 코드로 트랜스파일링)

        • 최신 ECMAScript 환경을 만들기 위해 코드가 실행되는 환경에 존재하지 않는 빌트인, 메소드 등을 추가하는 역할.

  • 웹팩이란? 모듈 번들러란? rollup
    • 모듈 번들러
      • 웹 애플리케이션을 구성하는 자원 (HTML, CSS< JS, Images 등)을 각각의 모듈로 보고 이를 조합해 병합된 하나의 결과물을 만드는 도구
      • 모듈 : 프로그래밍 관점에서 특정 기능을 갖는 작은 코드 단위 웹팩에서의 모듈 : 웹 애플리케이션을 구성하는 모든 자원을 의미 (HTML, JS, Font 등) 모듈 번들링 : 웹 애플리케이션을 구성하는 많은 자원들을 하나의 파일로 병합/압축해주는 동작
      • 등장 배경 (아래 문제점 해결)
        • 자바스크립트 변수 유효 범위 ⇒ 파일 단위의 JS 모듈 관리
        • 브라우저별 HTTP 요청 숫자의 제약
        • 사용하지 않는 코드의 관리
        • Dynamic Loading & Lazy Loading 미지원 빠른 로딩 속도와 높은 성능 필요 → 브라우저에서 서버로 요청하는 파일 숫자 줄이기. 웹 태스크 매니저 이용해 파일 압축/병합. 레이지 로딩 등장(초기 페이지 로딩 속도 up)
        • 웹 개발 작업 자동화 도구 필요 (코드 수정 후 새로고침, 자원 압축, CSS 전처리기 변환) ⇒ Grunt, Gulp 도구 등장
      • 장점
        • 여러 파일의 JS 코드를 압축해 최적화 가능하기 때문에 로딩에 대한 네트워크 비용 절감됨
        • 모듈 단위로 개발 가능. 가독성 유지보수 쉬움
        • 최신 JS 문법을 지원하지 않는 브라우저에서 사용 가능한 코드로 쉽게 변환시켜 줌

    • webpack
      • 오픈 소스 JS 모듈 번들러
      • 최신 FE 프레임워크에서 가장 많이 사용됨
      • Entry, Output, Loaders, Plugins, Mode로 구성
        • Entry : 웹팩이 빌드할 파일의 시작 위치 나타냄
          • entry 지점으로부터 import 되어있는 다른 모듈과 라이브러리에 대한 의존성 찾음
          • 기본 값 : ./src/index.js
        • Output : 웹팩에 의해 생성되는 번들을 내보낼 위치와 파일 이름 지정
          • 기본 값 : ./dist/main.js
        • Loaders : JS 파일이 아닌 파일들도 유효한 모듈로 변환시켜 줌
          • 설정은 싱글 모듈에 대한 rules 프로퍼티를 정의해야 함 rules 프로퍼티는 test와 use를 필수 프로퍼티로 가지고 있음
          • test 프로퍼티 : 변환해야하는 파일 또는 파일들을 식별하는 역할
          • use 프로퍼티 : 변환되어야하는 파일에 대해 어떤 로더를 사용해야 하는지 설정
        • Plugins : 로더가 파일단위로 처리하는 반면 플러그인은 번들된 결과물을 처리함
          • 번들된 JS를 난독화, 특정 텍스트 추출하는 용도로 사용 가능
        • Mode : 웹팩 세팅 시 development, production, none 중 하나를 파라미터로 사용
          • production : 최적화되어 빌드
          • development : 빠르게 빌드
          • none : 아무 기능 없이 웹팩으로 빌드
      • 정리

        웹팩은 entry로 설정된 시작점에서 의존성을 가진 모든 파일을 압축해 output 지점에 하나의 자바스크립트 파일을 만들어 줌 이때, 자바스크립트가 아닌 파일은 loaders를 이용하여 자바스크립트에서 이용 가능한 모듈로 만들어 주며, plugins를 이용하여 번들된 자바스크립트를 난독화하거나 특정 텍스트를 추출하는 역할을 함. mode는 웹팩의 사용 목적에 따라 설정을 지정하는 역할을 함

  • npm과 yarn은 어떻게 다른가요?
    • 공통점
      • Node.js의 패키지 관리자
      • 전 세계 개발자들이 JS로 만든 다양한 패키지를 npm 온라인 DB에 올리면 npm, yarn 같은 패키지 관리자를 통해 설치 및 삭제 가능함
      • CLI를 통해 패키지 설치, 삭제 뿐 아니라 패키지 버전 관리, 의존성 관리도 편하게 가능
    • npm
      • Node Package Manager (노드 패키지 매니저)
      • 노드를 설치 시 자동 설치되는 기본 패키지 관리자
      • 온라인 플랫폼으로서 노드 패키지를 만들고, 업로드하고, 공유할 수 있는 공간으로 누구나 온라인 플랫폼 (npm 레지스트리)에 게시된 패키지 사용 가능함
      • 온라인 플랫폼과 상호 작용하기 위해 명령 줄 인터페이스를 사용해 패키지 설치 밎 체거 가능
    • yarn
      • 2016년 페이스북에서 개발한 패키지 관리자
      • 리액트(React)와 같은 프로젝트를 진행하며 겪었던 어려움을 해결하기 위해 개발됨
      • npm 레지스트리와 호환하면서 속도나 안정성 측면에서 npm보다 향상됨
      • npm을 통해 설치
    • 차이점
      • 속도
        • 패키지 설치 프로세스를 처리하는 방법 서로 다름
        • npm : 패키지를 한번에 하나씩 순차적으로 설치
        • yarn : 여러 패키지를 동시에 가져오고 설치하도록 최적화 (npm보다 빠름)
      • 보안
        • yarn이 더 안전한 것으로 알려져 있음
        • npm은 자동으로 패키지에 포함된 다른 패키지 코드를 실행함 ⇒ 보안 시스템에 몇 가지 취약석 발생하며 나중에 심각한 문제 발생 여지 있음
        • yarn은 yarn.lock 또는 package.json 파일에 있는 파일만을 설치함
        • 최근 npm 업데이트에서 npm의 보안 업데이트도 크게 향상됨
      • 명령어
    • 결론

      둘 다 모두 종속성 관리하고 패키지 관리하기 좋은 툴. 지속적으로 관리되고 있으며 폭넓은 사용자 커뮤니티 가지고 있고 업데이트를 통해 추가된 기능 덕에 거의 차이나지 않음. 개인의 취향, 성능(패키지 설치 속도), 커뮤니티에 따라 선택하면 됨
